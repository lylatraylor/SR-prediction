# Serve Receive Set Prediction — 2025 Season

A machine learning project exploring offensive predictability in collegiate women's volleyball. Built as part of my work as a graduate assistant and statistician for the UC Berkeley women's volleyball program.

---

## Project Overview

In volleyball, the serving team has a significant advantage if they can anticipate who the opposing setter will distribute the ball to before the play develops. This project asks the question:

> **How predictable was our offense in serve receive during the 2025 season — and did that change over the course of the year?**

Rather than framing this purely as a model accuracy problem, the goal is to use a trained classifier to generate probability distributions over potential set targets for each point, and use those distributions to measure offensive predictability by rotation, pass quality, and season period.

The broader research question this project is designed to prototype is:

> *For scouting purposes, can a model trained on an opponent's recent matches learn who they are most likely to set in each rotation given certain conditions, and use that to make real-time predictions during a game?*

---

## Data

The primary data source for this project is **live match coding** conducted by the author courtside using VolleyStation throughout the 2025 season. This live-coded data captures real-time play-by-play information including rotation, pass quality, set destinations, and attack outcomes as they occur.

This live data was supplemented and cross-referenced with post-game `.dvw` files (DataVolley scouting format) provided by **VolleyMetrics**. The VolleyMetrics files are richer in detail — capturing additional coded attributes not always feasible to record live — and were used in this project to explore a broader feature space during development.

Feature selection was informed by a combination of domain knowledge and variable reduction principles. An initial broader set of variables available in the `.dvw` format was considered, with features ultimately selected based on what is both statistically meaningful and realistically capturable during live game coding. This reflects the practical constraint that any deployed version of this system would depend on live-coded inputs rather than post-game files.

The 2025 season comprises 30 matches divided into three periods:

| Period | Games | Training | Testing |
|---|---|---|---|
| Preseason | 10 | Games 1–8 | Games 9–10 |
| First Half Conference | 10 | Games 11–18 | Games 19–20 |
| Second Half Conference | 10 | Games 21–28 | Games 29–30 |

Each period has its own independently trained model. Training on the first eight games of each period and testing on the final two reflects the realistic scouting scenario — using accumulated knowledge from the season to that point to predict behavior in the most recent matches, which tend to be played in the same weekend.

**Note on data scope:** This project uses only our own team's data, both for ease of access and out of respect for opponent data privacy. Results and interpretation are therefore framed as a retrospective self-analysis rather than an opponent scouting application.

## Methodology

### Data Extraction (R)

`extract_serve_receive.R` uses the [`datavolley`](https://github.com/openvolley/datavolley) R package to parse `.dvw` files and produce a clean CSV with one row per serve-receive point. Each row captures:

- Server and receiver identity and evaluation codes
- Pass quality (0–4 numeric scale)
- Setter, set code, and set type
- Attacker identity, attack code, and outcome
- Receiving team rotation (derived from setter position using American convention)
- Full lineup for both teams (home/visiting player IDs by position)
- Score differential, set number, and point outcome

### Feature Engineering (Python)

Feature selection prioritized variables that are both predictively meaningful and realistically available during live game coding. The full `.dvw` format contains many more coded attributes than are included here; this set reflects a deliberate reduction toward features with clear volleyball domain justification.

| Feature | Description |
|---|---|
| `receiving_rotation` | Rotation 1–6 (setter position mapped to American convention) |
| `pass_quality` | Numeric pass quality (0–4) |
| `score_diff` | Score differential from focus team's perspective |
| `set_number` | Which set of the match (1–5) |
| `server_encoded` | Identity of the serving player |
| `recv_p1–6_enc` | Receiving team's six players by court position |
| `opp_p2–4_enc` | Opponent's front row blockers |
| `player_top_attack_enc` | Each attacker's most frequent attack type (derived from training data only) |
| `prev_point_won` | Whether the focus team lost the immediately preceding point in the dataset |
| `prev_attacker_enc` | Who was set on the immediately preceding point, if team was in serve-receive |

**Lag feature scope:** `prev_point_won` and `prev_attacker_enc` are scoped to consecutive serve-receive points only — detected by checking whether the current `point_id` is exactly one greater than the previous row's within the same set and game. If a side-out occurred between serve-receive sequences, both features reset to a neutral value rather than carrying over irrelevant context.

### Modeling

Three separate **XGBoost multi-class classifiers** are trained — one per season period. Each model outputs a probability distribution over potential set targets for a given point (e.g. 50% outside hitter, 30% middle, 15% right side, 5% back row attacker).

A **valid attacker mask** is applied at prediction time: probabilities for players not on the court in a given rotation are zeroed out and the remaining probabilities are renormalized. This ensures the model only considers players who are actually available to be set on that play.

Encoders for categorical features are fit on training data only and applied consistently to test data, with unseen labels handled gracefully rather than causing errors. Attack code profiles — each player's most frequent attack type — are derived from training data only to avoid leakage into test evaluation.

### Predictability Analysis

For each point, the model's masked probability distribution is used to compute:

- **Shannon entropy** — higher entropy indicates the model sees a less concentrated, less predictable distribution across hitters
- **Top prediction accuracy** — how often the model's highest-probability pick matched what actually happened
- **Non-top choice rate** — how often the setter chose someone other than the model's top prediction

These metrics are computed for all passes and separately for in-system passes (quality ≥ 3), since out-of-system offense is inherently more constrained and predictable regardless of tendency. Comparing these two conditions is a core part of the analysis.

## Key Limitations & Future Work

**Live integration:** A fully deployed version of this system would update predictions in real time as a match is coded live. While direct integration with VolleyStation has not been explored in this project, it is not necessarily out of reach — one plausible approach would be embedding a lightweight model in a worksheet environment and applying Gauss-Markov updating as new points are coded. This is noted as a meaningful direction for future work rather than a closed problem.

**Data richness vs. live availability:** The `.dvw` files used here contain more coded detail than is feasible to capture during live game coding. Feature selection in this project was deliberately guided by what a live statistician could realistically record, making the feature set more conservative than the full `.dvw` format would allow. The tradeoff between model richness and live deployability is a central design tension in this work.

**Bayesian updating:** A natural extension would replace or augment the static XGBoost model with a Bayesian updater — using the trained model's output as a prior and updating it point-by-point as new evidence accumulates during a match. This is the theoretically correct formulation for the live scouting problem and is planned as a next step.
